## 1. Что такое «Школа ИКС»

«Школа ИКС — Центр знаний» — это **образовательная экосистема**, а не просто сайт.
Она объединяет три направления обучения и пять пользовательских ролей в одном пространстве.

### 1.1 Три направления обучения

| Направление | Slug | Аудитория | Что внутри |
|---|---|---|---|
| Школа | `school` | 1–11 класс | репетиторство, помощь с ДЗ, индивидуальный план |
| IT-академия | `academy` | дети и подростки | Python, Web, Unity, проекты и портфолио |
| ОГЭ / ЕГЭ | `exam` | 9 и 11 класс | подготовка к экзаменам, пакеты занятий |

**Публичный сайт — единая витрина** без переключателя «режимов»: все три направления показаны как обычные карточки в блоке `#directions`. На лендинге нет `data-mode` / `body[data-mode]` — устаревшая логика удалена. Узнать направление лида можно только из формы заявки (поле `direction` обязательно).

Слой темизации `body[data-mode]` остаётся в `tokens.css` как опциональная функция кабинетов, если им потребуется акцент по предмету выбранного ребёнка. На публичной странице не используется.

### 1.2 Пять ролей (MVP)

1. **Гость** — видит публичный сайт, оставляет заявку через `XSchool.addLead`. Не видит цены, пакеты, кабинеты, расписание, ДЗ, статистику.
2. **Родитель** — оплачивает, видит детей, расписание, ДЗ, остатки пакетов, историю списаний и платежей, статистику. Не меняет правила биллинга — действует единое правило 8 часов.
3. **Ученик** — видит свои уроки и ДЗ, сдаёт работы (текст + до 5 файлов: jpg/png/pdf, ≤10 МБ/файл).
4. **Преподаватель** — проводит уроки (индивидуальные и групповые), проверяет ДЗ, отмечает посещаемость, работает в CRM.
5. **Администратор / Директор** — центр управления школой: люди, финансы, обучение, настройки тарифов и услуг. Реализован в `admin.html`, демо-вход `+79990000000 / 0000`.

## 3. Доменная модель (сущности)

Все сущности живут в `localStorage` (MVP, без бэкенда). Единый namespace `xschool.*`.

### 3.1 User
```
{
  id, role: "parent" | "student" | "teacher",
  phone, name, avatar?, createdAt
}
```

### 3.2 Family (связка)
```
{
  parentId, studentIds: [id, id, ...]
}
```
Связка строится **по номеру телефона родителя**, указанному при записи ученика.

### 3.3 Student (расширяет User)
```
{
  ...User,
  grade,            // класс
  modes: ["school" | "academy" | "exam"],
  subjects: [subjectId],
  parentPhone
}
```

### 3.4 Teacher (расширяет User)
```
{
  ...User,
  subjects: [subjectId],
  bio, photo, rating?
}
```

### 3.5 Subject
```
{
  id, mode, title, icon, colorAccent
}
```

### 3.6 Lesson (единица расписания и биллинга)
```
{
  id, studentId, teacherId, subjectId,
  startAt, durationMin,
  status: "planned" | "tomorrow" | "done" | "missed" | "cancelled",
  topic?, notes?, createdAt, updatedAt
}
```
Статус `tomorrow` — производный (вычисляется), но кэшируется для скорости рендера.

### 3.7 Package (пакет уроков)
```
{
  id, studentId, subjectId,
  lessonType: "individual" | "group",
  total, used, remaining,
  pricePerLesson, totalPaid,
  startedAt, endsAt?
}
```
> Поле `policy` отсутствует. Действует единое правило списания из `06_BILLING.md` (окно 8 часов).

### 3.8 Payment
```
{
  id, parentId, packageId?,
  amount, method, status: "paid" | "pending" | "failed",
  createdAt, comment?
}
```

### 3.9 Charge (списание урока с пакета)
```
{
  id, lessonId, packageId, studentId,
  amount: 1,           // обычно 1 урок
  reason: "lesson_done" | "lesson_missed_strict" | "manual",
  createdAt
}
```
Списание = транзакция. Никогда не уменьшаем `package.remaining` без записи `Charge`.

### 3.10 Homework
```
{
  id, lessonId, studentId, teacherId, subjectId,
  title, description, deadline,
  status: "assigned" | "in_progress" | "submitted" | "checked" | "needs_revision",
  studentAnswer?, teacherComment?, score?,
  createdAt, updatedAt
}
```

### 3.11 Notification — УДАЛЕНО ИЗ MVP

В MVP сущности `Notification` нет. Вся ключевая информация отражена в:
- расписании (модалка отмены/переноса с дедлайном 8h);
- ДЗ (статусы и дедлайн-бейджи);
- пакетах (баннеры «Низкий остаток» / «Пакет закончился» на главной);
- истории списаний и платежей.

В будущем сюда вернётся on-demand генератор + inbox, если потребуется.

## 4. Ключевые сценарии (E2E)

Каждый сценарий проверяется по 5 контрольным вопросам:
1. Что видит **родитель**?
2. Что видит **ученик**?
3. Что видит **преподаватель**?
4. Как это влияет на **оплату**?
5. Как это влияет на **расписание**?

### 4.1 Запись на пробное (гость → лид)
- Гость заполняет форму на публичной странице (имя, телефон, направление, предмет/комментарий).
- Создаётся **`Lead`** через `XSchool.addLead({ name, phone, mode, direction, subject, source: 'landing', status: 'new' })`.
- Никаких сущностей `Parent`/`Student`/`Lesson` на этом шаге **не создаётся** — это работа администратора/директора.
- Заявка попадает в `admin.html → Люди → Заявки`.
- Администратор по заявке может: сменить статус (`new → in_progress → booked / rejected`), создать `Parent`, создать `Student`, далее назначить преподавателя и создать `Lesson` через CRM.
- Публичная страница говорит «Пробное занятие — бесплатно». Бесплатность отражается на этапе создания первого `Lesson` (без `Charge`) или решается администратором при оформлении.

### 4.2 Проведение урока
- Преподаватель открывает CRM → отмечает `done` / `missed` / `cancelled`.
- При `done` → биллинг создаёт `Charge`, `package.remaining -= 1`.
- При `cancelled` → урок переносится, списания не происходит.
- При `missed` → решение зависит от `package.policy`.
- Ученик и родитель видят обновление статуса в расписании.

### 4.3 Цикл ДЗ
- Преподаватель выдаёт ДЗ при создании/проведении урока → статус `assigned`.
- Ученик открывает → `in_progress`. Сдаёт → `submitted`.
- Преподаватель проверяет → `checked` или `needs_revision` (с комментарием).
- Родитель видит финальный статус и комментарий.

### 4.4 Оплата пакета
- Родитель выбирает пакет → создаётся `Payment` (`pending`) → подтверждение → `paid`.
- На `paid` → создаётся `Package`, `remaining = total`, прогресс-бар обновляется.

### 4.5 Низкий остаток
- При `remaining ≤ 2` → уведомление родителю «Пора продлить пакет».
- При `remaining = 0` → блокировка создания новых уроков преподавателем (мягкая, с предупреждением).

## 5. Правило автосписания (единое)

Полная спецификация — в `06_BILLING.md`. Кратко:

- **Списывается:** урок проведён; либо ученик не пришёл и отмена не была сделана минимум за 8 часов до начала занятия.
- **Не списывается:** отмена за 8+ часов, либо отмена со стороны преподавателя/школы.
- Для групповых уроков решение принимается **по каждому ученику отдельно**.

Принцип: **никаких тихих списаний**. Каждое списание = запись `Charge` с готовым `reasonText`,
видимая в истории родителя.

## 6. Уведомления — УДАЛЕНО ИЗ MVP

Раздел сохранён как историческая справка. Ни типы, ни триггеры в MVP не реализованы.
В будущем сюда вернётся on-demand генератор + inbox.

Старая (нереализованная) таблица типов:

| Тип | Кому | Триггер |
|---|---|---|
| `lesson_tomorrow` | родитель + ученик | за 24ч до урока |
| `lesson_done` | родитель | после `done` |
| `homework_assigned` | родитель + ученик | новое ДЗ |
| `homework_checked` | родитель + ученик | проверено |
| `package_low` | родитель | `remaining ≤ 2` |
| `package_empty` | родитель | `remaining = 0` |
| `payment_success` | родитель | `paid` |
| `student_submitted_hw` | преподаватель | `submitted` |

## 7. Технический стек (MVP)

- **Frontend:** HTML + CSS + Alpine.js
- **Хранилище:** `localStorage` (namespace `xschool.*`)
- **Состояние:** Alpine `$store` + reactive обёртка над `localStorage`
- **Роутинг:** хеш-роутинг (`#/parent`, `#/student`, `#/teacher/...`)
- **Адаптивность:** mobile-first, breakpoints `640 / 1024 / 1280`
- **Темизация:** в `tokens.css` определены `body[data-mode="academy"]` и `body[data-mode="exam"]` для опционального акцента в кабинетах. Публичный сайт `data-mode` НЕ использует.

Бэкенд — **вне MVP**. Архитектура данных спроектирована так, чтобы в будущем заменить `localStorage` на REST/Supabase без изменения UI-слоя.

## 10. Принципы, которые нельзя нарушать

1. **Прозрачность денег.** Любое списание видно родителю с причиной.
2. **Связка по телефону.** Родитель ↔ ученик связываются по `phone`, а не по почте.
3. **Один статус — одно действие.** Не пытаемся «угадать» статус: преподаватель ставит явно.
4. **Mobile-first.** Каждый кабинет должен работать на телефоне родителя в очереди в магазине.
5. **Не ломать визуальный язык.** Все ЛК наследуют токены и темы публичного сайта.
6. **Документация — первична.** Сначала меняем `docs/`, потом код. После — запись в `09_CHANGELOG.md`.
