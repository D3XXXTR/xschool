## 2. User Flows

### Flow 1. Запись на занятие (гость → лид → обработка админом)

```
гость на /
   │
   ▼
жмёт «Записаться» (hero | направление | эксперт | локация)
   │
   ▼
открывается одна универсальная lead-модалка
   ├─ имя
   ├─ телефон
   ├─ направление *  ← обязательно (school / academy / exam)
   └─ предмет / комментарий
   │
   ▼
submit → XSchool.addLead({
   name, phone,
   mode: direction, direction,
   subject, source: 'landing',
   status: 'new'
})
   │
   ▼
плашка «Заявка сохранена. Мы свяжемся с вами в ближайшее время.»

──────── ниже работа администратора в admin.html ────────

admin → Люди → Заявки
   ├─ просмотр lead с фильтром (new / in_progress / booked / rejected)
   ├─ изменение статуса
   ├─ «Создать родителя» из заявки (по phone)
   └─ «Создать ученика» (привязка к родителю)
   │
   ▼
Назначения (admin → Люди → Назначения)
   └─ привязка преподавателя к ученику по предмету
   │
   ▼
Создать урок (admin → Обучение → Расписание → +Урок)
   └─ Lesson(planned)
```

**Принцип:** публичная форма создаёт **только Lead**. Никаких `Parent`/`Student`/`Lesson` напрямую. Это разделяет публичную лидогенерацию и оперативную работу школы.

**Edge cases (на стороне админа):**
- Телефон родителя уже есть в системе — повторное «Создать родителя» делает дедуп: использовать существующего и привязать к нему ученика.
- Если из заявки нельзя достоверно понять имя ребёнка — админ создаёт черновую запись и уточняет вручную.

### Flow 3. Типичный день родителя

```
открывает /parent (главная)
   │
   ▼
видит «ближайший урок» + «дедлайн отмены без списания»
   │
   ├── всё ок → закрывает
   │
   ├── надо отменить →
   │      ▼
   │      жмёт «Отменить» рядом с уроком
   │      ▼
   │      [подтверждение: до dl осталось Xч]
   │      ▼
   │      lesson.cancelledAt=now, cancelledBy=parent
   │      ▼
   │      биллинг: sees ≥8h → не списывать;
   │                ELSE → списать (видно в истории + сигнал в баннере)
   │
   └── надо продлить пакет →
          ▼
          /parent/billing/buy (мастер из 5 шагов)

          Payment(paid) → Package создан → toast «+8 уроков»
```

### Flow 4. Типичный день ученика

```
открывает /student (Сегодня)
   │
   ▼
видит ближайший урок + ДЗ на сегодня
   │
   ├── есть ДЗ →
   │     ▼
   │     открыть → читает задание
   │     ▼
   │     status: assigned → in_progress
   │     ▼
   │     заполняет ответ (текст + файлы)
   │     ▼
   │     [Сохранить черновик] / [Сдать на проверку]
   │     ▼
   │     submitted → счётчик «К проверке (N)» в sidebar преподавателя
   │
   ├── урок через час →
   │     ▼
   │     [Войти в урок] (для онлайн — ссылка/Zoom; MVP: показать заглушку)
   │
   └── ничего срочного → закрывает
```

### Flow 5. Типичный день преподавателя

```
открывает /teacher (Сегодня)
   │
   ▼
видит уроки на день, ближайший подсвечен
   │
   ├── провести индивидуальный урок →
   │     ▼
   │     [Провести] → форма: тема, заметки, статус (• Проведён)
   │     ▼
   │     [+ Выдать ДЗ] (опц.)  → создаётся Homework(assigned)
   │     ▼
   │     [Сохранить] → биллинг: Charge(lesson_done) → −1 урок у ученика
   │
   ├── провести групповой урок →
   │     ▼
   │     [Открыть урок] → форма со списком участников
   │     ▼
   │     по каждому ученику: ( done / missed / cancelled )
   │     ▼
   │     [Сохранить]
   │     ▼
   │     биллинг идёт по каждому участнику (см. §7 в 06_BILLING.md):
   │       done           → Charge(lesson_done)
   │       cancelled (≥8ч)→ skip
   │       cancelled (<8ч)→ Charge(lesson_missed_late)
   │       missed         → Charge(lesson_missed_late)
   │
   ├── проверить ДЗ →
   │     ▼
   │     /teacher/homework → список «к проверке»
   │     ▼
   │     открыть → оценка + комментарий
   │     ▼
   │     ( • Принять )    → status=checked
   │     ( ○ Доработать ) → status=needs_revision
   │     ▼
   │     статус ДЗ обновляется в карточке ученика и родителя
   │
   └── создать урок →
         ▼
         /teacher/new-lesson (5 шагов)
         ▼
         Lesson(planned) появляется у ученика и родителя в расписании
```

## 3. Спецификация ключевых UX-элементов

### 3.1 Прогресс-бар пакета

**Состояния:**
| Условие | Цвет | Бейдж |
|---|---|---|
| `remaining > 30% от total` | зелёный | — |
| `remaining ≤ 30%` | жёлтый | — |
| `remaining ≤ 2` | красный | ⚠ Низкий остаток |
| `remaining = 0` | серый + красный баннер сверху ЛК | Закончился |

Подпись всегда: `{used} из {total}` + текст-помощник: `{remaining} {урок|ов} осталось`
(склонение: 1 урок / 2-4 урока / 5-20 уроков, дальше по правилам ru).

### 3.2 История списаний — карточка (read-only)

```
[icon]  [амт] [—подпись—]
         {reasonText}        ← всегда начинается с «Урок списан: …»
         {created_at в человеческом виде}
```

Иконки:
- ✓ зелёная — `lesson_done`
- ⚠ жёлтая — `lesson_missed_late`
- + фиолетовая — пополнение (Payment → Package)

Оспаривания в системе нет (см. `06_BILLING.md` §6).

Сортировка: новое сверху. Бесконечный скролл; на мобиле — пагинация по 20.

### 3.3 Плашка «Урок завтра» (без отдельного inbox)

В MVP отдельных уведомлений нет. На «Главной» родителя и ученика показывается
data-driven плашка ближайшего урока. Её содержимое:
- Название предмета, имя ребёнка/преподавателя, время урока.
- **Конкретное локальное время дедлайна отмены без списания** (= `startAt − 8h`),
  не «за 8 часов».

Кнопка действия — «Изменить» (открывает модалку отмены/переноса). Триггеров,
inbox и push-уведомлений нет.

### 3.4 Групповые уроки — список участников + статусы

**Карточка в расписании:**
- Заголовок: `ГРУППА · {Subject} · {time}`.
- Имя преподавателя.
- Список участников с индивидуальным статусом (точка-цвет + ярлык).
- Если своих детей в группе несколько — все подсвечены, остальные обезличены.
- В CRM преподавателя — все имена и переключатели статуса (см. E3 в `02_SITE_STRUCTURE.md`).

**Правила отображения статуса:**
- ⚪ planned / tomorrow
- ✓ done
- ⚠ missed (с подписью «без отмены за 8ч» если так)
- ✕ cancelled (с подписью кто и когда отменил)
- 🔵 «текущий пользователь» подсвечивается рамкой

**Биллинг:** одна строка `Charge` на каждого участника, кому положено списание (см. `06_BILLING.md` §4–§7). В истории родителя такие списания группируются заголовком урока.

### 3.5 Низкий остаток

Триггеры:
- `remaining ≤ 2` → бейдж на карточке + жёлтый баннер на главной ЛК родителя.
- `remaining = 0` → красный баннер вверху ЛК до момента покупки нового пакета.

Текст: «Пакет заканчивается. Чтобы не пропустить уроки — продлите.» Кнопка `[ Продлить ]` ведёт в `/parent/billing/buy` с предзаполненным выбором (тот же ребёнок и предмет).
